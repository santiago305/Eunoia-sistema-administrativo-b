# Auditoria inicial - Notifications/Messaging (2026-05-11)

## Backend notifications (estado actual)
- Existe modulo funcional de notificaciones simples basado en `notifications` + `notification_recipients`.
- Servicio principal: `application/use-cases/notifications.service.ts`.
- Entrada HTTP actual: listado, detalle, mark seen/read, mark all read, archive.
- Realtime ya implementado por websocket (`notification.gateway.ts`) + service (`notification-realtime.service.ts`).
- Outbox y delivery attempts ya existen para entrega realtime robusta.

## Frontend notifications (estado actual)
- Feature minima: listado y detalle simple.
- Realtime frontend activo via `NotificationProvider` + `sileo` toasts.
- No existe UX tipo Gmail: inbox/sent/drafts/trash/starred/thread/compose flotante.

## Integraciones detectadas
- Compras ya dispara eventos de notificacion en varios casos de uso/controladores usando `createNotificationForUsers`.
- `sourceModule: "purchases"` ya se esta enviando desde compras.
- Permisos actuales en access-control: `notifications.read` y `notifications.manage`.

## Permisos y filtros por modulo
- No existe aun endpoint de modulos permitidos para sidebar.
- No existe validacion granular de `originModule` por usuario en consultas de notificaciones.

## Que se puede reutilizar
- Canal websocket actual.
- Integracion frontend con Sileo para alerta visual.
- Pipeline de generacion de notificaciones desde compras.
- Infraestructura base de modulo y estructura general.

## Que se debe reemplazar/ampliar
- Modelo de datos: crear `messages`, `message_recipients`, `message_attachments`, `message_threads`, `message_search_history`, `message_audit_logs`.
- Casos de uso para mensajeria corporativa (send, drafts, reply, forward, trash, starred, bulk, search history).
- Endpoints nuevos orientados a carpetas y detalle por hilo.
- Control de permisos por `originModule`.
- Frontend completo de mensajeria tipo Gmail.
